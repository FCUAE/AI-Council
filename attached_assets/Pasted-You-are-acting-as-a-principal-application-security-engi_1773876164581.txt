Task #64
Title:
Security: Fix dangerous account reassignment on email collision

Problem:
The current auth sync path can overwrite ownership by rewriting the primary user id on email collision.

Required fix:
- In server/replit_integrations/auth/storage.ts, remove any logic that updates an existing user's primary key based on email collision.
- Never transfer credits, conversations, Stripe customer linkage, or billing state between identities automatically.
- Safe behavior must be:
  - Same Clerk user id: normal upsert/update
  - Same email but different Clerk user id: fail closed, do not merge, do not rewrite ids
- Do not silently continue as if sync succeeded.

Implementation detail:
- Return a typed result from upsertUser, such as:
  - success
  - email_collision_blocked
  - user_not_found
- Do not return a normal success object for a blocked collision
- In server/replit_integrations/auth/replitAuth.ts, update syncClerkUser to explicitly handle blocked collision state
- Log a structured security event for collisions with:
  - event type
  - email
  - old id prefix
  - new id prefix
- Do not log full ids or sensitive data

Acceptance details:
- Existing user row ownership is never reassigned
- Same-id returning users still work
- New legitimate users still work
- Email collision cannot take over another user's data
- Collision handling is explicit and safe

Task #65
Title:
Security: Protect uploads and object storage with auth and ACL

Problem:
Sensitive debate attachments and support uploads are publicly accessible or insufficiently protected.

Required fix for /uploads:
- Stop treating /uploads/:filename as a public file host for sensitive user content
- Debate attachments and support images must be private by default
- Require authentication before serving sensitive uploaded files
- Enforce server-side ownership checks before serving
- Unauthorized access must return generic 403 without disclosing file existence details

Important implementation requirement:
Do not implement ownership checks by scanning all conversations inefficiently on each request if it can be avoided.
Prefer a durable server-side ownership model such as:
- a file metadata table, or
- explicit attachment-to-owner mapping, or
- object ACL-based ownership persisted at upload time

Required fix for /objects:
- In the /objects route, resolve user identity
- Enforce ACL through canAccessObjectEntity before download
- Public objects may be served if ACL allows public read
- Private objects must require auth plus ACL validation
- Do not call downloadObject before access control passes

Support upload hardening:
- POST /api/support/upload must require authentication
- Support uploads should be private by default
- Support file access must be bound to authenticated ownership or admin access only

Do not:
- Leave a public fallback route for old sensitive files
- Leak object paths or upload filenames in denial logs

Acceptance details:
- Unauthenticated users cannot access private uploads or private objects
- Authorized owners still can
- ACL is actually enforced, not just checked superficially
- Support uploads are no longer an unauthenticated public file host

Task #67
Title:
Security: Strengthen file upload validation

Problem:
Upload validation currently relies too much on MIME and extension and not enough on actual file content.

Required fix:
- Implement magic-byte or file-signature validation for allowed upload types
- Validate extension, MIME, and content signature together
- Reject mismatches
- Reject suspicious filenames including:
  - double extensions
  - control characters
  - null-byte style tricks
  - dangerous active-content extensions
- Explicitly block:
  - svg
  - html
  - exe
  - js
  - shell scripts
  - any active or executable content
- TXT should only be accepted if it does not look binary
- DOCX validation should acknowledge that ZIP header confirms container type, not full semantic validity
- Keep the allowed file type list as narrow as possible

Document parsing hardening:
- Tighten error handling in server/documentParser.ts
- Ensure all temp files are cleaned in try/finally
- Add timeouts and size or page limits where needed
- Limit extracted text length safely
- Avoid parser crashes from bubbling sensitive internals to clients

Serving behavior:
- Non-image downloads should use attachment disposition
- Sensitive files should not be inline-served unless explicitly safe and intended

Acceptance details:
- Renamed malicious files are rejected
- Mismatched extension, MIME, and signature are rejected
- Active content is blocked
- Legitimate supported files still upload successfully
- Parsing failures fail safely without leaking internals
