You are an expert front end engineer and UI designer. Your task is to refactor and redesign my existing website in this Replit project to use the IBM Carbon Design System as the primary UI framework.

High level goal  
- Keep the existing content, pages, and general user flows of my site.  
- Completely modernize the UI using Carbon Design System components, layout, and design tokens so that the site looks and feels like a native Carbon product experience.  
- Code should be clean, modular, and production ready.

Tech stack and libraries  
- Use React with TypeScript as the main implementation (if the project is not already React, migrate the UI layer to React without breaking functionality).  
- Install and use the official Carbon React implementation from the Carbon monorepo: `@carbon/react`.  
- Install Carbon styles and tokens from the monorepo, for example `@carbon/styles` and the theme helpers if needed.  
- For icons, use the Carbon icon components provided by the Carbon packages instead of random SVGs.  
- Do NOT introduce design systems like Material UI, Tailwind, Bootstrap, or other component libraries. Carbon should be the single design system.

(If the current project is not React and cannot be migrated reasonably, then instead use the Carbon Web Components package `@carbon/web-components` to implement the UI with framework agnostic custom elements. In that case, still follow all layout, spacing, and theming guidance from Carbon.)

Design reference and documentation  
- Use the official Carbon Design System documentation as your source of truth for UX and UI decisions:  
  - Components, patterns, and usage guidelines  
  - Layout and grid  
  - Spacing, typography, and color tokens  
  - Theming (white, gray 10, gray 90, gray 100)  
- Use the official React tutorial and demo repos only as reference for structure and best practices, not as copy paste:  
  - Main Carbon monorepo: https://github.com/carbon-design-system/carbon  
  - React + Next.js tutorial app: https://github.com/carbon-design-system/carbon-tutorial-nextjs  
- Match the overall feel to typical Carbon based IBM interfaces: clean, minimal, dense but not cramped, with strong hierarchy and accessible contrast.

Information architecture and layout  
- Analyze all existing pages, routes, and sections in this project.  
- Create a clear layout structure using the Carbon grid and layout primitives:  
  - Use the 12 column grid with responsive breakpoints as Carbon recommends.  
  - Replace ad hoc flex and custom grids with Carbon grid where possible.  
  - Use Carbon `Content` and `Grid` patterns to constrain width and align content.  
- Introduce a Carbon based global layout:  
  - Use Carbon UI Shell components (header, side nav, skip to content, etc) if appropriate for a multi page app.  
  - Add a top navigation bar with Carbon `Header` and `HeaderNav` components including logo, main navigation items, and any key actions (for example Sign in or Contact).  
  - For apps with many sections, use a left side navigation using Carbon `SideNav`.  
- Make sure the layout is responsive on mobile, tablet, and desktop following Carbon breakpoints. Test the layout down to narrow widths.

Components and patterns to use  
For every current UI element, replace it with the closest Carbon component while keeping the same function and content.

Examples of what I expect:  
- Buttons  
  - Replace all custom buttons with Carbon `Button` and `ButtonSet`.  
  - Use the primary, secondary, tertiary, and danger variants where appropriate.  
- Forms and inputs  
  - Use Carbon `TextInput`, `TextArea`, `NumberInput`, `Select`, `Dropdown`, `ComboBox`, `Checkbox`, `RadioButtonGroup`, and `Toggle` instead of custom styled inputs.  
  - Add labels, helper text, and validation text using Carbon patterns.  
- Lists and data display  
  - Use `DataTable`, `StructuredList`, `Tag`, `Chip`, `Accordion`, `Tabs`, `Tiles`, and `Card` like patterns where needed.  
  - For tables, support sorting, empty states, and loading states following Carbon examples.  
- Feedback and overlays  
  - Use Carbon `InlineNotification`, `ToastNotification`, `Modal`, `Loading`, and `Skeleton` states.  
- Navigation and page structure  
  - Use `Breadcrumb` at the top of content pages when hierarchy exists.  
  - Use `Tabs` for sectioned content instead of custom tab code.  
  - Use `Pagination` where there are long lists.  
- Icons  
  - Replace all random or inconsistent icons with official Carbon icons imported from the Carbon icon package.  
  - Choose icons that match Carbon usage guidance and are consistent across the site.

Theming, tokens, and visual system  
- Pick a Carbon theme that best suits a modern product website, for example Gray 10 or Gray 90, and apply it consistently.  
- Use Carbon design tokens for:  
  - Colors (for example interactive, background, text, border tokens)  
  - Spacing scale (use the spacing tokens rather than random pixel values)  
  - Typography (IBM Plex families, sizes, and weights as Carbon defines)  
  - Radius and elevation where appropriate  
- Replace hard coded colors, fonts, and spacing in CSS with Carbon tokens and mixins.  
- Ensure color contrast meets accessibility guidelines using token pairs that Carbon recommends.

Code structure and architecture  
- Organize the React code into a clear component hierarchy:  
  - `components/` for reusable Carbon wrapped components (Button groups, page sections, layouts).  
  - `layouts/` or similar folder for layout shells using Carbon UI Shell and grid.  
  - `pages/` or route components that assemble these building blocks.  
- Encapsulate Carbon styling logic in a single Sass or CSS entry that imports Carbon styles and themes, rather than scattering imports.  
- Avoid deeply nesting custom styled components when Carbon already provides a pattern. Favor composition of Carbon components instead.  
- Remove unused CSS files and redundant styles that Carbon replaces.  
- Ensure all TypeScript types are correct, and add prop types where necessary.

Accessibility  
- Follow Carbon accessibility guidance by default.  
- Make sure all interactive elements are keyboard accessible, focusable, and properly labeled with `aria` attributes when needed.  
- Attach labels correctly to form elements and always provide helper or error text when validation is present.  
- Use semantic HTML structure under the Carbon components (headings order, landmarks like main, nav, footer).

Migration method  
- Step 1: Scan the current codebase to understand the existing routes and UI.  
- Step 2: Set up Carbon in the project, including dependencies and global styles.  
- Step 3: Implement the global layout (header, navigation, content shell) using Carbon components.  
- Step 4: For each page, refactor the markup to use Carbon components while preserving functionality and data flow.  
- Step 5: Replace remaining custom controls with Carbon versions.  
- Step 6: Clean up unused styles, components, and assets.  
- Step 7: Run the app, test each page, and fix visual or functional regressions.

Deliverables  
- A working version of this site in Replit with:  
  - Carbon based layout and navigation.  
  - Carbon components used across all pages instead of custom UI.  
  - Design tokens applied for colors, typography, and spacing.  
  - Clean, documented code. Add comments where the structure or token usage may not be obvious.  
- A short summary in comments at the root (for example in README or a main component) describing:  
  - Which Carbon theme is used.  
  - Where to look for main layout components.  
  - How to add new pages or components while staying consistent with Carbon.
