Council Brief: Make Everything 10x Better SECTION 1: PROMPT ENGINEERING (The Core Product) Current System Prompts: Stage 1 (Hearing): "You are a member of an AI Council. Provide your best answer to the user's query. Be direct, clear, and helpful." Stage 2 (Review): "You are a peer reviewer on an AI Council. Give a concise, focused critique of your colleagues' responses. Use bullet points. Be brief but thorough." Stage 3 (Chairman Verdict): "You are the Chairman of the AI Council. Synthesize a final, comprehensive answer based on the initial opinions, peer review feedback, and your own analysis. Address any issues raised in the reviews. Be clear, direct, and actionable. Do not use markdown formatting - write in plain text with clear paragraphs." Questions for the Council: These prompts are generic. How should we specialize them so the council actually debates rather than just giving polite summaries? How do we get real intellectual friction? The review stage truncates peer responses to 2,000 characters. Is that too short for complex topics? What's the ideal balance between context and focus? The Chairman is told "no markdown." Should we reconsider this? Our frontend can render styled text. Would structured output (headers, bullets, code blocks) improve readability? Should each council member get a distinct persona or perspective (e.g., "You are the skeptic," "You are the optimist," "You are the domain expert") to produce genuinely diverse viewpoints? For follow-up messages, previous context is prepended as raw text. Should we use a more structured format? Should the summary (generated by GPT-4o-mini after turn 3) be tuned differently? SECTION 2: ARCHITECTURE & RELIABILITY Current Architecture: Express backend, React frontend, PostgreSQL (Drizzle ORM) OpenRouter as LLM gateway to 37 models Clerk auth, Stripe payments, Resend email Reserve-then-settle credit system (10% buffer, refund on completion) Polling-based status updates (1s interval) Known Failure Modes: If the server crashes mid-deliberation, reserved credits are never refunded (no recovery mechanism) Retry logic could theoretically spawn two concurrent deliberation processes if the abort signal is slow If a model fails in Stage 1, the review stage still runs with incomplete data No circuit breaker — if OpenRouter is down, all 3 models fail sequentially with 5-min timeouts each Questions for the Council: 6. Should we add a "credit recovery" cron that finds conversations with settled = 0 older than 30 minutes and auto-refunds? What's the right timeout? Should we switch from polling (1s) to Server-Sent Events (SSE) for real-time deliberation updates? What are the tradeoffs at scale? How should we handle partial failures (1 of 3 models fails)? Currently we proceed with 2/3. Should we substitute a backup model? Show the user a degraded-but-complete result? Should we add a circuit breaker pattern for OpenRouter? If 3 consecutive calls fail, stop trying for N minutes? SECTION 3: CREDIT SYSTEM & MONETIZATION Current Pricing: COST_PER_CREDIT = $0.05 Explorer: 100 credits / $15 ($0.15/credit) Strategist: 325 credits / $39 ($0.12/credit) Visionary: 900 credits / $89 ($0.098/credit) Free users: 15 credits, limited model selection Credits expire after 60 days with a 30-day warning email How Credits Are Calculated: Based on estimated input/output tokens for 3 council members + chairman Attachment tokens increase cost (images=1000 tokens, PDFs=500*(size/50000)) Reserve = ceil(cost * 1.1), settle = ceil(reserve / 1.1), refund difference Questions for the Council: 10. Is the pricing competitive? A typical 3-model debate with GPT-4o costs ~2-3 credits ($0.10-0.15). Premium models like GPT-5.4 Pro cost ~8-12 credits ($0.40-0.60). Are we leaving money on the table or pricing too high? Should we offer a subscription model (monthly credits that reset) in addition to one-time packs? The 60-day expiration feels aggressive. What's the right balance between urgency and fairness? Should purchased credits never expire? Should follow-up messages cost less than initial debates since they reuse context? SECTION 4: UX & FRONTEND Current Experience: Landing page with rotating hero text, 3-step explanation cards Model selection via popover with champion badges, cost indicators, vision tags Debate display: DebateGrid with individual model cards → ChairmanCard verdict → EvidenceVault tabs (Individual Opinions / Cross-Examination / Performance Rankings) "Did You Know" fact rotator during 30-90 second processing Sidebar with conversation history, credit balance Questions for the Council: 14. The debate takes 30-90 seconds. Is the current waiting experience (progress bar + rotating facts + model card animations) engaging enough? What would make users feel the wait is worth it? Should users be able to see the debate happening in real-time (streaming individual model responses) rather than seeing them all at once when complete? The EvidenceVault "Performance Rankings" parses review text for sentiment. Is this gimmicky or genuinely useful? How could we make model comparison more meaningful? Should we add a "Quick Mode" that skips the debate (Stage 2) and just gives 3 independent answers + a synthesis? Half the wait time, lower cost. SECTION 5: SCALING & PERFORMANCE Current Bottlenecks: Server-side document extraction runs in batches of 4 (recently fixed from unbounded) PDF rendering to images (pdftoppm) is CPU-intensive and synchronous All 3 LLM calls + chairman call happen sequentially within the pipeline (Stage 1 is parallel, but Stage 2 waits for Stage 1, Stage 3 waits for Stage 2) No caching — identical prompts always trigger fresh deliberations No queue system — deliberations run directly in the Express process Questions for the Council: 18. Should we add response caching for identical or near-identical prompts? What similarity threshold would be appropriate? At what user count should we move deliberation processing to a job queue (Bull/BullMQ with Redis)? What are the signs we're hitting the limit? Should we pre-warm popular model combinations so the first API call to OpenRouter is faster? SECTION 6: CASE STUDIES (Stress Tests) Case Study 1: 30 Medical PDFs + GPT-5.4 Pro Council User uploads 30 medical reports (avg 50 pages each) Total extracted text: ~3M characters, truncated to 200K Each PDF extraction: ~2-5 seconds Total deliberation: ~90 seconds Cost: ~12 credits ($0.60) Recently fixed: parallel upload batching, per-file truncation, model context window caps Case Study 2: Image-Heavy Architecture Review User uploads 15 building blueprints as PDFs PDFs have no extractable text → rendered as images (15 images, hitting the cap) Vision-capable models analyze images; text-only models get "[Note: you don't have vision]" Risk: non-vision models contribute less useful analysis Case Study 3: Quick Coding Question User asks "How do I sort an array in Python?" 3 models + chairman = ~$0.01 in API costs but costs 1 credit ($0.05) 30-60 second wait for a trivial question Should this type of question even go through the full council process? Questions for the Council: 21. For Case Study 1 (document-heavy), is 200K chars enough context? Should we implement smarter extraction (summarize each document first, then send summaries)? For Case Study 2 (image-heavy), how should we handle mixed vision/non-vision councils? Should non-vision models be auto-swapped for vision models when images are the primary input? For Case Study 3 (trivial questions), should we detect question complexity and route simple questions to a single fast model instead of the full council? SECTION 7: COMPETITIVE POSITIONING Questions for the Council: 24. Our unique value prop is "multiple AI models debating to find the best answer." How do we communicate this better? Current tagline rotates: "most intelligent advice/strategy/code..." What features would make power users (developers, researchers, consultants) choose this over just using ChatGPT/Claude directly? Should we add model comparison features — showing which model was "most helpful" across a user's history? Should we open-source the debate protocol and let people self-host?
