### The Markup Formula Correction

**I'm siding with GPT-5.4 Pro's margin-derived formula over the Chairman's cost-band markup table.** The cost-band approach was elegant but mathematically inconsistent with its own stated target. Here's the corrected approach:

```typescript
function computeCreditCharge(
  estimatedApiCost: number,
  failureBuffer: number = 0.08,     // 8% for retries/failures (see Section below)
  targetMargin: number = 0.55,
  worstCaseNetPerCredit: number = 0.0957,  // Visionary after Stripe
  costPerCredit: number = 0.05,
  minCredits: number = 3
): number {
  // Total cost including failure risk
  const bufferedCost = estimatedApiCost * (1 + failureBuffer);
  
  // How many credits needed to hit target margin at worst-case revenue?
  // margin = (revenue - cost) / revenue
  // 0.55 = (credits × 0.0957 - bufferedCost) / (credits × 0.0957)
  // credits × 0.0957 × 0.45 = bufferedCost
  // credits = bufferedCost / (0.0957 × 0.45)
  // credits = bufferedCost / 0.04307
  
  const rawCredits = bufferedCost / (worstCaseNetPerCredit * (1 - targetMargin));
  
  // Round to nearest "nice number" for UX
  const rounded = niceRound(rawCredits);
  
  return Math.max(minCredits, rounded);
}

function niceRound(credits: number): number {
  if (credits <= 5) return Math.ceil(credits);        // 1, 2, 3, 4, 5
  if (credits <= 20) return Math.ceil(credits / 2) * 2;  // 6, 8, 10, 12...
  if (credits <= 50) return Math.ceil(credits / 5) * 5;  // 25, 30, 35...
  return Math.ceil(credits / 10) * 10;                    // 50, 60, 70...
}
```

**Now let's re-price Scenario D:**
- Estimated API cost: $0.184
- Buffered: $0.184 × 1.08 = $0.199
- Raw credits: $0.199 / 0.04307 = 4.62
- Nice-rounded: **5 credits**
- Margin check: 5 × $0.0957 = $0.479 revenue, minus $0.184 cost = 62% margin ✓

**At 5 credits per debate, free users with 50 credits get ~10 debates.** That's a proper conversion funnel.

## The Depth Mode Question: Ship Standard-Only for V1

**I side with Claude Opus 4.6: skip depth mode for V1. Ship a single "standard" mode with fixed output caps.**

Gemini pushed hard for depth mode, arguing it's essential for reasoning models and that users expect control. But Gemini's own cross-examination undermined this argument. ## Reasoning Model Handling: The Real Threat

**This is the single most important unresolved issue from the cross-examination.** Both Gemini and Claude flagged it. Neither proposed a complete solution. Here's mine:

### The Problem

Reasoning models (o1, o3, DeepSeek R1) produce "thinking tokens" that are:
1. **Billed as output tokens** by most providers
2. **Sometimes not reported** in `usage.completion_tokens`
3. **Not controllable** via `max_tokens` on some providers (the cap applies to visible output only; thinking is uncapped)

This breaks every assumption in the pricing formula. Your `max_tokens: 2500` cap might produce 2,500 visible tokens plus 8,000 thinking tokens, and you've budgeted for 2,500 total.

### The Decision

**For V1, restrict reasoning models to the Chairman slot only, and apply a reasoning-token risk multiplier to the Chairman cost estimate.**

```typescript
// Model configuration includes a reasoning flag
interface ModelConfig {
  id: string;
  apiCostInput: number;   // per 1M tokens
  apiCostOutput: number;  // per 1M tokens
  isReasoningModel: boolean;
  reasoningTokenMultiplier: number; // 1.0 for standard, 2.5-4.0 for reasoning
}

// In estimateDebateCost, modify Stage 3 calculation:
const chairmanOutputEstimate = Math.round(s3Out * UTIL);
const chairmanOutputForCost = chairmanModel.isReasoningModel
  ? chairmanOutputEstimate * chairmanModel.reasoningTokenMultiplier
  : chairmanOutputEstimate;
totalCost += calcCost(chairmanModel, chairmanInput, chairmanOutputForCost);
```

**Starting multipliers (conservative, tuned with data):**

| Model | `reasoningTokenMultiplier` | Rationale |
|-------|---------------------------|-----------|
| o1 | 3.0 | Heavy thinking, moderate visible output |
| o3 | 3.5 | Extended thinking, often even more verbose internally |
| DeepSeek R1 | 2.5 | Thinking shown in output, slightly more predictable |
| All standard models | 1.0 | No hidden tokens |

**Why restrict reasoning models to Chairman only in V1?** Because in Stage 1 and Stage 2, you have 6 calls with unpredictable thinking costs. One bad estimate is absorbable; six bad estimates compound. The Chairman slot is a single call, and even with a 3.5× multiplier, the cost is bounded and estimable. Users can still select reasoning models for their Chairman — they'll just see a higher credit price and understand why.

**Gemini's UTIL = 1.15 for reasoning models is directionally correct but insufficiently aggressive.** A utilization factor above 1.0 doesn't capture the real problem, which is that thinking tokens can exceed visible tokens by 3-5×. A dedicated multiplier on the cost calculation is cleaner and more honest.

## Failure-State Economics: The Gap Nobody Filled

**Claude Opus 4.6's cross-examination correctly identified this as the biggest missing piece.** Every council member modeled the happy path. Here's the unhappy path:

### Failure Taxonomy

| Failure Type | Frequency (est.) | Cost Impact | User Impact |
|-------------|-------------------|-------------|-------------|
| API 429 (rate limit) | 2-5% of calls | Retry cost = 1× call cost | Delay |
| API 500 (provider error) | 1-2% of calls | Retry cost = 1× call cost | Delay |
| Malformed response (JSON parse fail) | 1-3% of calls | Retry cost = 1× call cost | Delay |
| Timeout (>30s response) | 0.5-1% of calls | Wasted cost, no output | Delay or abort |
| Full pipeline failure (>2 retries fail) | <0.5% of debates | All prior call costs wasted | Full abort |

### The Policy (Ratified)

```typescript
const RETRY_CONFIG = {
  maxRetriesPerCall: 1,        // ONE retry, not infinite
  retryDelayMs: 2000,          // 2-second backoff
  abortAfterConsecutiveFailures: 2, // If 2 calls in a row fail, abort the debate
};

async function executeDebatePipeline(config: DebateConfig): Promise<DebateResult> {
  const callResults: CallResult[] = [];
  let totalActualCost = 0;
  
  for (const call of config.calls) {
    let result = await executeCallWithRetry(call, RETRY_CONFIG);
    
    if (!result.success) {
      // ABORT: Refund ALL credits. User pays nothing for a failed debate.
      await refundCredits(config.userId, config.creditCharge);
      await logFailedDebate(config.debateId, callResults, result.error);
      
      // If partial results exist (some stages completed), STORE them
      // but don't charge. User can see them but gets credits back.
      if (callResults.length > 0) {
        await storePartialResults(config.debateId, callResults);
      }
      
      return { success: false, partialResults: callResults, refunded: true };
    }
    
    totalActualCost += result.actualCost;
    callResults.push(result);
  }
  
  // Success: settle at quoted price, log actuals
  await settleCredits(config.userId, config.reserved, config.creditCharge);
  await logActualUsage(config.debateId, callResults, totalActualCost);
  
  return { success: true, results: callResults };
}
```

**Key decisions:**
- **Full refund on failure.** Not partial credit, not "sorry." Full refund. The trust cost of a partial charge on a broken debate is far higher than eating $0.10-0.50 in sunk API costs.
- **Store partial results.** If Stages 1-2 completed but Stage 3 (Chairman) failed, the user can still see the council opinions. This turns a failure into a "degraded but useful" experience.
- **The 8% failure buffer in the pricing formula covers this.** The `failureBuffer: 0.08` in `computeCreditCharge` is specifically to absorb retry costs and occasional full-abort losses across the user base. At a 0.5% full-failure rate and 3-5% retry rate, 8% buffer is sufficient. Track it and adjust.
