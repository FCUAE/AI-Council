Issue 1
Critical: Object ACL bypass in /api/uploads/extract-text

Problem
The app now enforces ACL on /objects/*, but /api/uploads/extract-text still reads object-storage files directly with getObjectEntityFile and processes them without calling canAccessObjectEntity. This creates an alternate path to access private object content.

Files likely involved
- server/routes.ts
- server/replit_integrations/object_storage/objectStorage.ts
- server/replit_integrations/object_storage/objectAcl.ts

Required fix
- In every /api/uploads/extract-text path that touches object storage, resolve the current authenticated user first
- Before reading the object, call objectStorageService.canAccessObjectEntity with READ permission
- If access is denied, return 403 with a generic message
- Log the denied attempt using securityLogger without exposing the raw object path
- Apply this for both:
  - image object paths
  - document object paths
- Do not allow extraction, metadata reading, page counting, rendering, or temp-file creation before ACL passes

Acceptance
- A user cannot extract text or metadata from another user's private object
- Public objects still work if public read is explicitly allowed
- Denied access is logged safely
- No object content is processed before authorization succeeds

Issue 2
High: /uploads/:filename still allows access to files with no ownership record

Problem
The new /uploads route checks ownership only if an owner record exists. If there is no owner record, the file is still served to any authenticated user. This likely affects:
- legacy uploads created before file_uploads tracking
- failed tracking inserts
- generated PDF page images or other derived files

Files likely involved
- server/replit_integrations/object_storage/routes.ts
- server/routes.ts
- server/documentParser.ts
- database migration logic if needed

Required fix
- Change /uploads/:filename to fail closed when ownership metadata is missing
- Only serve a file if one of these is true:
  - the requester is the recorded owner
  - the requester is admin
  - the file is explicitly marked public in a safe and intentional way
- Do not treat missing ownership as safe
- If needed, add a stronger file metadata model to support:
  - filename
  - owner user id
  - purpose
  - public/private flag
  - parent resource id if needed
  - generated-from source if needed

Generated file handling
- Review all server-generated files such as rendered PDF page images
- Do not leave generated artifacts in uploads without ownership tracking
- Best option:
  - keep generated images internal only and never serve them through /uploads
- Acceptable fallback:
  - record them in file_uploads or a better metadata table with the correct owner and purpose
  - ensure cleanup happens safely

Legacy compatibility
- If old files already exist without ownership metadata, do not leave them readable
- Add a migration strategy or documented fallback for old files
- If migration is required, document it clearly in SECURITY_AUDIT.md

Acceptance
- Missing ownership means deny access, not allow access
- Generated files are no longer orphaned
- Legitimate owners still access their files normally
- Admin access still works where intended

Issue 4
High: CSRF protection is only partial

Problem
State-changing middleware checks Origin only when the header is present. Missing-Origin requests are currently allowed. This is not a complete defense for browser-facing authenticated mutation routes.

Files likely involved
- server/index.ts
- server/routes.ts
- any auth/session handling related code

Required fix
- Tighten CSRF protection for browser-facing state-changing authenticated routes
- Do not silently allow missing Origin on sensitive browser routes unless there is a clearly verified reason
- Best options:
  1. require a valid Origin for sensitive browser mutation routes, or
  2. implement a real CSRF token mechanism, or
  3. combine strict Origin checking with route-specific CSRF protection
- At minimum, sensitive routes like these must be covered carefully:
  - DELETE /api/user
  - Stripe mutation routes
  - conversation creation and add message
  - retry and cancel routes
  - support routes if authenticated
  - profile update routes

Important nuance
- Some legitimate non-browser flows may not send Origin
- If so, separate those routes clearly instead of leaving browser-sensitive routes broadly permissive
- Do not reduce security just to avoid edge-case breakage

Acceptance
- Sensitive browser mutation routes reject unauthorized cross-origin requests
- Missing-Origin behavior is handled intentionally, not passively
- CSRF-like attempts are logged
- Normal in-app usage still works

Issue 5
Medium: Rate limiting and abuse protection are still too weak for real production

Problem
Per-user throttling is currently in-memory, not distributed, and not route-scoped enough. This is not reliable in multi-instance production and may create weird cross-route throttling.

Files likely involved
- server/routes.ts
- server/index.ts
- storage or database if durable locks or counters are added

Required fix
- Replace or significantly improve the in-memory perUserBuckets system
- Prefer route-scoped and durable throttling using a shared store if available
- At minimum, separate limits by route category, not only by user id
- Add explicit concurrency protection for expensive debate processing
- Limit active debates per user in a way that survives multiple processes if possible
- Add retry cooldown for retry endpoints
- Add stronger protection for:
  - conversations.create
  - conversations.addMessage
  - conversations.retry
  - uploads.direct
  - uploads.extract-text
  - support
  - support.upload
  - stripe.recover-credits
  - stripe.sync-credits

Important
- Do not make limits so strict that normal users break
- Keep the UX reasonable
- If only a partial distributed solution is possible now, document the limitation in SECURITY_AUDIT.md

Acceptance
- Abuse is harder for a single authenticated user
- Retry spam is controlled
- Active processing amplification is controlled
- Route limits are scoped sensibly
- Document any remaining multi-instance limitation if not fully solved
