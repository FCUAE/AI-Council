Act as a senior application security engineer and secure code reviewer. Your job is to perform a full security audit and security hardening pass across this entire project, identify every realistic security weakness, fix what can be fixed safely and with low risk, and clearly report what you changed, what remains risky, and what requires manual decisions.

This must be a true full-project audit, not a limited pass based only on the first issues you find.

Core requirements:
- Preserve all existing functionality
- Do not break current flows, integrations, UI behavior, or business logic
- Be aggressive in finding issues, but conservative in risky code changes
- Apply all safe fixes directly where possible
- If a fix is risky or architecture-changing, explain it and recommend the safest path instead of forcing a breaking rewrite
- Think like both a defender and an attacker
- Assume this app may become production-facing and must be hardened accordingly
- Do not stop after checking only auth, only dependencies, or only the obvious routes
- Review the whole project thoroughly
- For any area where you find no issue, explicitly state that you checked it and why it appears safe

Scope:
Review the entire codebase and all relevant security surfaces, including:
- frontend code
- backend code
- shared utilities
- auth flows
- authorization logic
- admin endpoints
- user-scoped endpoints
- API routes
- server actions
- database queries and ORM usage
- file uploads, downloads, previews, parsing, and storage
- Stripe and payment-related logic
- Clerk authentication integration
- OpenRouter or other AI provider integrations
- session and token handling
- environment variable usage
- secrets handling
- build configuration
- deployment and Replit-related configuration
- infrastructure-related files
- webhook handling
- logging and observability paths
- background jobs, cron tasks, workers, and queues
- object storage and signed file access
- tests, scripts, migrations, and seed logic
- dependency manifests and lockfiles

Objectives:
1. Find security vulnerabilities
2. Fix them directly where safe and low risk
3. Harden the application against common attack paths
4. Preserve app functionality exactly
5. Explain all findings, fixes, and any remaining risks
6. Continue the audit beyond the initially discovered findings until every major security category has been checked

Follow this exact security review checklist:

A. Secrets and sensitive data
- Search the entire project for hardcoded secrets, API keys, tokens, passwords, private URLs, credentials, signing keys, and connection strings
- Move secrets to environment variables if needed
- Ensure secrets are never exposed to frontend bundles, logs, client responses, or public files
- Check that .env files, secret files, build outputs, and generated artifacts are properly ignored by git
- Verify no sensitive values are committed in config files, source code, test files, fixtures, documentation, scripts, or examples
- Review whether any frontend environment variables expose sensitive configuration accidentally
- Flag any secrets that cannot be rotated automatically

B. Authentication and authorization
- Review login, signup, logout, password reset, email verification, session handling, token issuance, refresh tokens, magic links, OAuth flows, Clerk integration, and role-based access
- Ensure authentication checks cannot be bypassed
- Ensure every protected route, API endpoint, admin page, background job, and privileged action enforces authorization server-side
- Audit all privileged and user-scoped endpoints for missing server-side authorization, object ownership checks, and tenant isolation
- Check for insecure direct object reference issues, broken access control, privilege escalation, and missing multi-tenant boundaries
- Verify users can only access their own data unless explicitly allowed
- Confirm admin-only functionality is truly restricted on the backend, not just hidden in the UI
- Review all admin routes, not just the first exposed one you find

C. Session and token security
- Verify cookies use HttpOnly, Secure, SameSite where appropriate
- Check token expiration, refresh logic, revocation, replay resistance, and storage safety
- Ensure tokens are not stored insecurely in localStorage unless absolutely necessary and clearly justified
- Review JWT or session signing configuration for weak secrets, unsafe algorithms, missing validation, or overly permissive trust boundaries
- Ensure session fixation risks are addressed
- Invalidate sessions on logout and sensitive account changes where possible
- Verify whether cookie-based or Clerk-based auth needs CSRF protections on state-changing actions

D. Input validation and output handling
- Review all request inputs, query params, path params, headers, form inputs, uploaded files, webhooks, AI prompts where relevant, and database inputs
- Add strict validation and sanitization where missing
- Prevent SQL injection, NoSQL injection, command injection, template injection, path traversal, header injection, unsafe deserialization, and mass assignment
- Prevent reflected, stored, and DOM-based XSS
- Ensure output encoding is context-aware
- Validate all file names, MIME types, extensions, sizes, counts, and storage paths for file uploads
- Prevent browsers from unsafe inline rendering of uploaded files where that creates risk

E. API and backend security
- Audit every API route and server action for auth, authorization, validation, rate limiting, and safe error handling
- Ensure no internal stack traces, secrets, SQL errors, raw provider errors, or sensitive debug details leak to clients
- Verify unsafe HTTP methods are restricted where appropriate
- Add protections against over-posting, mass assignment, and trust in client-provided privileged values
- Ensure webhooks verify signatures and reject untrusted origins where applicable
- Verify CORS is minimally permissive and not wildcarded unless absolutely required
- Check SSRF risks in any URL fetch, image proxy, scraper, import, remote file feature, webhook callback flow, or external fetch path
- Review helper functions that construct absolute URLs and ensure they are not vulnerable to Host header injection

F. Database and data protection
- Review all database queries, ORM usage, raw SQL, migrations, and seed logic
- Ensure parameterized queries or safe ORM patterns are used everywhere
- Check row-level access patterns and multi-user data boundaries
- Flag overly broad queries that may leak data
- Ensure sensitive fields are not returned unnecessarily
- Check encryption or hashing for passwords and highly sensitive data
- Verify passwords use a modern password hashing algorithm such as bcrypt, argon2, or equivalent with safe parameters
- Ensure no plaintext passwords are stored or logged

G. Frontend and client security
- Check for exposed environment variables in the frontend
- Prevent XSS in rendered user content, markdown, HTML injection, dangerouslySetInnerHTML, template rendering, and URL handling
- Review local storage, session storage, client state, and browser caches for sensitive data leakage
- Ensure forms and client calls do not expose internal implementation details
- Check unsafe redirects and open redirect risks
- Review CSP compatibility and recommend or apply safe improvements
- Ensure no auth or authorization assumptions rely only on hidden UI states

H. Dependency and supply chain security
- Audit all dependencies and lockfiles for known vulnerabilities, abandoned packages, suspicious packages, unnecessary packages, postinstall risk, and overly broad version ranges
- Run dependency vulnerability checks and review critical and high severity findings
- Update dependencies safely where possible
- Remove unused risky dependencies if they are not needed
- Check postinstall scripts and custom scripts for risky behavior
- Flag packages with known severe CVEs, suspicious maintenance history, or supply chain red flags

I. Security headers and browser protections
- Review and improve security headers including Content-Security-Policy, X-Frame-Options or frame-ancestors, X-Content-Type-Options, Referrer-Policy, Permissions-Policy, and Strict-Transport-Security where applicable
- Check CSRF protections for cookie-based auth and state-changing requests
- Ensure clickjacking protections are in place
- Check mixed content and insecure asset loading risks
- Ensure user-served files and uploads are not vulnerable to browser MIME sniffing

J. File handling and storage
- Audit file upload, download, preview, parsing, import, export, and attachment flows
- Prevent executable file upload risks, path traversal, malicious SVG or HTML issues, oversized file abuse, excessive file count abuse, and unsafe temp file handling
- Ensure uploaded files are stored outside executable paths where applicable
- Verify signed URLs or access controls for private file access
- Validate attachment count and total size server-side, not just client-side
- Decide clearly whether public upload endpoints should remain public and be heavily hardened, or require authentication. Do not leave this vague.

K. Logging, monitoring, and error handling
- Ensure logs do not contain secrets, tokens, personal data, passwords, payment information, raw auth headers, or full sensitive request bodies
- Review error handling and observability hooks for leakage
- Ensure production mode disables unsafe debug behavior
- Replace unsafe console logging of sensitive objects
- Add safer structured logging or redaction where needed

L. Rate limiting, abuse, and denial of service
- Identify endpoints vulnerable to brute force, credential stuffing, spam, scraping, enumeration, payment abuse, upload abuse, expensive parsing abuse, and denial of service
- Add rate limiting, cooldowns, pagination limits, request size limits, timeouts, body size limits, and concurrency protections where relevant
- Prevent user enumeration in auth, recovery, or support-related flows
- Check expensive endpoints and unbounded queries
- Review credit recovery, sync, payment setup, portal access, and related billing flows for replay, abuse, race conditions, duplicate processing, or trust in client-side values

M. Infrastructure and configuration
- Review Dockerfiles, deployment configs, CI/CD configs, GitHub Actions, Replit settings, package scripts, build config, startup config, host handling, trusted proxies, and HTTPS assumptions
- Remove dangerous defaults
- Ensure production uses secure settings
- Check environment separation between development and production
- Ensure dev-only tooling is not exposed in production
- Flag missing operational hardening issues visible from the repo

N. Business logic security
- Look for workflow abuse, price tampering, coupon abuse, payment manipulation, race conditions, duplicate submissions, credit abuse, trust in client-side values, and privilege logic flaws
- Review onboarding, purchases, credits, invites, referrals, admin tooling, support flows, uploads, and user-generated workflows for abuse paths
- Ensure important actions are validated server-side and idempotent where needed

O. Privacy and compliance hygiene
- Minimize collection and exposure of personal data
- Ensure sensitive user data is not unnecessarily returned to the client
- Flag retention or exposure risks that are visible in the codebase
- Check cookie and tracking behavior for obvious privacy issues

Execution rules:
- First scan the repo and identify concrete findings
- Do not stop at the first batch of issues
- Continue checking the full checklist until every major category above has been reviewed
- Apply all safe, low-risk fixes directly in code
- If a fix could break functionality, explain the issue and propose the safest patch before making a major change
- Prefer secure defaults
- Prefer server-side enforcement over client-side checks
- Prefer explicit allowlists over blocklists
- Add validation schemas where appropriate
- Tighten types if that improves security
- Remove dead insecure code where safe
- Keep code readable and maintainable
- Where there is uncertainty about whether code is dead, sensitive, or relied on indirectly, do not remove blindly. Flag for manual review instead.

Testing requirements:
- After applying fixes, run the app and relevant tests
- Add or update tests for security-sensitive flows where feasible
- Verify auth, permissions, validation, uploads, billing-related flows, and critical business logic still work
- Verify admin access restrictions still work correctly
- Verify upload restrictions and file-serving behavior still work correctly
- Verify any Stripe or payment-related flows touched by fixes still function correctly
- Check that no secrets are printed during build or runtime
- Confirm the application still builds successfully
- Verify fixes do not break current UI/UX or expected user flows

Deliverables:
1. A security findings list before fixes
2. The fixes you plan to apply
3. The actual code changes
4. A final post-fix security report

1. A prioritized security audit summary with:
   - Critical

   - Low
2. For each finding include:
   - What the issue is
   - Where it exists
   - Why it matters
   - How you fixed it, or why it was not auto-fixed
3. A list of all files changed
4. Any follow-up recommendations that require product, architectural, or operational decisions
5. A final section called “Remaining Risks” for anything not fully resolved
6. A concise “Security Scorecard” covering:
   - auth
   - secrets
   - input validation
   - API security
   - dependency security
   - frontend security
   - infrastructure
   - abuse prevention
