SECTION 1: PROMPT ENGINEERING
1. Creating Real Intellectual Friction
Stop telling models to "debate." Instead, assign epistemic stances — not personality caricatures:

First Principles Analyst: Deconstructs assumptions, challenges the question's framing.
Pragmatic Implementer: Focuses on what works in practice, calls out theory that fails in the real world.
Contrarian Synthesizer: Finds the overlooked third option; if the other two would agree, this one dissents.
Each must produce: a thesis, stated assumptions, the strongest alternative view, failure modes, and a confidence level. Add the instruction: "If you agree with what a typical AI would say, you're not adding value." To prevent RLHF reversion to politeness, add structural guardrails: programmatically reject responses containing markers like "I agree with all points" or "It depends" without substantive follow-through.

2. Review Truncation (2,000 chars)
Too short. But raw length isn't the fix — structure is. Replace raw peer responses with a structured peer digest: 5 bullet claims + 2 quoted excerpts per peer, totaling ~4,000–6,000 characters. Make it dynamic: scale review context proportionally to Stage 1 response length. Prioritize preserving points of disagreement before truncation.

3. Chairman Markdown
Yes, enable markdown. The Chairman's synthesis is the primary deliverable — it must be the most readable element on the page. Use headers, bold for key conclusions, bullet points for actions, code blocks for technical content. Keep individual council members in lighter formatting so the Chairman feels like an upgrade.

4. Distinct Personas
Yes, but use methodological diversity, not personality labels. "Skeptic/Optimist/Expert" devolves into caricature. The epistemic stances above (First Principles / Pragmatist / Contrarian) produce genuine analytical diversity. Rotate or adapt by query category — coding questions need different lenses than business strategy. Let users choose council composition as a premium feature.

5. Stage 2 (Review) Prompts
This stage is where the most value is left on the table. Replace vague "critique politely" with:
> "Identify: (1) The strongest claim from a peer you disagree with — quote it and explain why. (2) A critical consideration ALL responses missed. (3) Which peer response you'd endorse if forced, and what it still gets wrong."

Forcing quotation compels engagement with actual content. Add a structural rule: each review must contain at least one substantive disagreement when warranted.

6. Chairman Synthesis
The Chairman must choose a side, not average opinions. Required output structure: (1) Decision + rationale, (2) Dissent notes, (3) Conditions under which the losing view would win, (4) Confidence level, (5) What evidence would change the recommendation. This is the "anti-averaging" principle — it produces sharper, more useful verdicts.

7. Follow-up Context & Summary
Raw text prepend degrades fast. Switch to structured context using XML/JSON:
```
<previousdeliberationsummary>[compressed summary]</previousdeliberationsummary>
<keyagreements>[bullets]</keyagreements>
<unresolveddisagreements>[bullets]</unresolveddisagreements>
<userfollowup>[new query]</userfollowup>
<instruction>Build on previous deliberation. Do not repeat settled points.</instruction>
```
For the GPT-4o-mini summary (after turn 3): tune it to be lossy in the right ways — preserve disagreements and open questions, omit redundant agreement. Use a compression schema: {decisions, contested_points, caveats, open_questions}. For high-value threads, use a stronger model for compression.

SECTION 2: ARCHITECTURE & RELIABILITY
8. Credit Recovery
Implement immediately. Use a state machine for deliberation status: reserved → processing → settled | failed | auto_refunded. Set timeout to 15 minutes for standard jobs (no legitimate deliberation takes that long), 2 hours for attachment-heavy jobs only if heartbeat is stale (>10 min since last heartbeat). Run cron every 5 minutes. Log every auto-refund. Notify users: "Your previous request encountered an issue. Credits refunded." Even better: if you implement BullMQ (see Q19), attach failure handlers directly to jobs — no cron needed.

9. Polling vs. SSE
Switch to SSE. Polling at 1s creates 60–90 requests per deliberation per client; SSE uses 1 persistent connection. SSE enables granular stage events (stage1_model1_complete, review_start, chairman_deliberating) which directly powers the streaming UX (Q15). Implementation: Redis pub/sub between workers and SSE endpoint for scale. SSE does have tradeoffs (proxy compatibility, mobile, connection management) — add polling as a silent fallback, not the primary transport. Per-model timeout: 45 seconds, not 5 minutes. Global wall clock: 120 seconds.

10. Partial Failure Handling
Use a quorum model: if 1/3 models fails, retry once with a 15-second budget. If retry fails, attempt one backup model (fast/cheap, e.g., GPT-4o-mini) within 10 seconds. If that also fails, proceed with 2/3 — but tell the user explicitly ("Model X failed; this deliberation includes 2 of 3 members"), adjust the review prompt accordingly, refund proportionally, and label the result "Degraded Consensus." If fewer than 2 succeed, abort entirely with full refund. Never silently mask failures.

11. Circuit Breaker
Yes. Implement per-model circuit breakers:

Closed (normal): calls go through.
Open (tripped): after 3 consecutive failures OR 50% failure rate in a 2-minute window, block calls for 60–120 seconds.
Half-open: after timeout, allow 1 probe call; success closes circuit, failure re-opens.
If GPT-4o is down but Claude works, only block GPT-4o. During open circuit, return an immediate honest error — don't make users wait for inevitable failure cascading across models.

SECTION 3: CREDIT SYSTEM & MONETIZATION
12. Pricing Competitiveness
Current pricing is reasonable for premium workflows but punitive for trivial queries. A GPT-4o debate at $0.10–0.15 is good value (3 perspectives + synthesis for ~2x a single ChatGPT message). GPT-5.4 Pro at $0.40–0.60 works for high-stakes use cases. Don't race to the bottom — race to the top on value. The fix isn't across-the-board price cuts; it's adding cheaper modes (Quick Mode, Single Expert) and smaller debit increments (0.25–0.5 credits for simple queries). Show estimated cost before each run.

13. Subscription Model
Yes, add subscriptions alongside credit packs:

Free: 15 credits, one-time (acquisition funnel)
Pro: $29/mo — 250–300 credits, rollover up to ~1 month's unused, priority processing
Team: $79/mo — 1,000 credits, shared pool, usage analytics
Keep credit packs for burst/occasional users
Subscriptions give predictable ARR, which is dramatically more valuable for fundraising and planning.

14. Credit Expiration
Purchased credits: never expire (or 12+ months minimum if legally required). Expiring paid credits feels like theft and generates chargebacks. Free/promotional credits: expire after 90 days. Subscription credits: expire at end of billing cycle + 30-day grace. Keep the 30-day warning email for any expiring credits.

15. Follow-up Pricing
Yes, charge less. Follow-ups reuse context and typically produce shorter outputs. Price at 50–70% of initial deliberation when no new files are attached. This encourages deeper conversations (stickier product) and reflects real cost savings from cached prompt prefixes. Advertise as "up to 40% cheaper." If context grows very long, the discount shrinks — base it on actual token delta, not a flat rule.

SECTION 4: UX & FRONTEND
16. Wait Experience (30–90s)
Current approach (progress bar + rotating facts + animation) is the minimum viable waiting experience. Improve by showing real pipeline progress via SSE: "Model 1 thinking → Model 2 responded → Cross-examination in progress → Chairman deliberating." Each stage transition is a dopamine hit. Show cost breakdown in real-time to reinforce value. Add a tab notification / sound so users can switch away. Kill generic "Did You Know" facts — they signal "we know this is too slow."

17. Real-time Streaming vs. All-at-Once
Stream it, stage-wise. This is the single highest-impact UX improvement. Show Stage 1 responses appearing in parallel (three cards filling). Brief pause ("Cross-examination beginning..."). Stage 2 reviews stream in. Chairman verdict streams last. Use stage-level streaming (show each card as it completes) rather than raw token-by-token, which causes cognitive overload. Collapsible "Focus mode" for users who prefer just the verdict.

18. Performance Rankings
Per-query sentiment parsing is unreliable and gimmicky. Replace with structured metrics: (1) Agreement map — which model's initial view was closest to Chairman's verdict, (2) Unique contribution — what each model added that others didn't, (3) User voting — thumbs up/down on individual responses after reading. Aggregate across queries for real value: "Over your last 50 questions, Claude was rated most helpful 62% of the time." Show per-category performance. Label per-query rankings as "experimental" if kept.

19. Quick Mode
Yes, add it. Three tiers:

Single Expert: 1 model, instant, cheapest (0.25–0.5 credits). For simple factual queries.
Quick Council: 3 models in parallel + Chairman synthesis, no cross-examination. ~40% faster, ~50% cheaper.
Full Council: Current 3-stage debate. For high-stakes decisions.
Auto-suggest mode based on complexity (use a lightweight classifier on prompt length, attachment presence, domain risk). Allow users to override. Add an "Escalate to Full Council" button on Quick results.

SECTION 5: SCALING & PERFORMANCE
20. Response Caching
Yes, but carefully:

Exact match only (same user, same prompt, same models): serve cached result, charge 0 credits, label "cached." Ask user: "Reuse previous answer?"
Do NOT cache across users. Context, attachments, and expectations differ.
No semantic similarity matching for cache hits — false positives erode trust. Exception: opt-in semantic cache with ≥0.92 cosine similarity for public, non-sensitive prompts only.
DO cache OpenRouter model metadata, pricing, and availability aggressively (5-min TTL).
Exclude: private conversations, attachments, high-risk domains (medical, legal). TTL: 24 hours for general knowledge.
21. Job Queue (BullMQ)
Implement now, before you need it. Don't wait for a threshold — the queue solves crash recovery (Q8), deployment safety, and horizontal scaling simultaneously. Architecture: Express API → enqueue job → return job ID → BullMQ worker (separate process) → runs deliberation → updates DB → SSE notifies client. Use unique job keys + DB advisory locks to prevent duplicate deliberations. Failed jobs trigger auto-refund handlers — cleaner than a cron. Signs you've waited too long: memory spikes, lost jobs during deploys, p95 > 2 minutes, CPU saturation from PDF rendering blocking the event loop.

22. Pre-warming Model Combinations
Low priority, low ROI. OpenRouter manages connection pooling and model warm state on their side. Your observed latency is primarily inference time, not connection overhead. What helps more: keep-alive HTTP connections to OpenRouter, cache model availability metadata, parallelize Stage 1 calls aggressively. Only invest in pre-warming if you measure cold-start spikes in your metrics — track p50/p95/p99 first-token latency per model before optimizing.

SECTION 6: CASE STUDIES
23. Huge Documents (30 Medical PDFs, 200K chars)
200K characters (~50K tokens) is insufficient for 30 PDFs at ~50 pages each. Implement a hierarchical ingestion pipeline: (1) Extract + OCR each document, (2) Generate a 1-page summary + key findings per document, (3) Build a retrieval index, (4) Send query-specific evidence packs (not full docs) to the council, with citations. All models must see the same evidence pack. For high-stakes medical content, add disclaimers and citation requirements to every prompt.

24. Vision/Image-Heavy Documents
Enforce capability checks. If >50% of context is image-based, auto-swap all council members to vision-capable models (e.g., swap Claude Haiku for Gemini Flash). Warn the user if their selected models can't handle the input. For mixed councils where some models lack vision, convert images to structured captions/OCR so text-only models can still contribute meaningfully. Never silently drop image content.

25. Trivial Questions
Implement a complexity router. Use a lightweight classifier (prompt length + attachment count + domain keywords, or a tiny model like Llama-3-8B scoring 1–10). If complexity < 3, route to Single Expert or Quick Council. If > 3, engage Full Council. Add an "Escalate to Full Council" button on quick results so users never feel locked out. This prevents the absurdity of running a 90-second, 3-credit debate on "What's the capital of France?"

SECTION 7: COMPETITIVE POSITIONING
26. Value Proposition
Lead with: "Three expert models argue. One verdict you can trust." Position against the "Confident Liar" problem of single LLMs — your product reduces hallucination through adversarial cross-examination. Show proof: disagreement rates, cases where peer review caught errors, before/after accuracy comparisons. Don't claim "smarter AI" — claim more reliable decisions.

27. Target Audience
High-stakes decision makers: developers evaluating architectures, legal professionals reviewing contracts, executives weighing strategic options, medical professionals seeking second opinions. These users care about accuracy and nuance over speed. Don't compete for casual chat users — that's a race to free.

28. Open Source
Open-source the debate protocol specification (how stages work, message formats). This drives developer adoption, ecosystem growth, and positions you as the standard. Keep proprietary: orchestration logic, system prompts, routing intelligence, performance data, and the tuned prompt templates that create genuine friction. Your moat is the quality of deliberation, not the concept of multi-model debate. Clones will appear regardless — brand, data flywheel, and prompt quality are your defensible advantages.

TOP 5 PRIORITIES (Implementation Order)
Fix prompts — epistemic stances + structured review rubric + anti-averaging chairman. Highest ROI, zero infra cost.
BullMQ job queue + credit recovery — reliability is table stakes for a paid product. Do before any marketing push.
SSE + stage-wise streaming — transforms a 60-second wait into a 60-second experience. The single biggest UX win.
Quick Mode + complexity router — captures trivial queries cheaply, reserves full debate for high-value questions. Fixes economics and UX simultaneously.
Subscription tier + fix credit expiration — predictable revenue + customer trust. Never expire purchased credits.

EVALUATION FRAMEWORK
Before changing multiple variables, instrument these metrics: disagreement rate (are models actually disagreeing?), chairman adoption rate (how often does the verdict track a specific member vs. averaging?), p95 latency, partial failure / degradation rate, credit margin per deliberation, cache hit rate, user satisfaction / thumbs-up rate, and retry rate. A/B test prompt variants before rolling them out universally.
