You are an expert front end engineer and UI designer.

My current stack
- React + TypeScript frontend
- Vite as the build tool
- Tailwind CSS with shadcn/ui for styling and components
- Express.js backend API that should remain unchanged

Goal
Redesign and refactor my existing React + Vite frontend so that it uses the IBM Carbon Design System as the primary design language and component library, while keeping:
- All existing routes, data flows, and API calls to the Express backend
- The same core content and user journeys

I want the site to look and feel like a Carbon based application, not a Tailwind or shadcn app.

Design system and libraries
- Use the official React implementation of Carbon Design System:
  - Install and configure `@carbon/react` as the main component library
  - Install and configure `@carbon/styles` for styles and Sass
  - Use Carbon tokens and helpers from packages such as `@carbon/themes`, `@carbon/layout`, `@carbon/type` and `@carbon/icons` where appropriate (check the main Carbon repo for the recommended packages list)
- Remove shadcn/ui components and Tailwind heavy styling in favor of Carbon components and Carbon design tokens
  - It is ok to keep a very thin layer of Tailwind utility classes only where Carbon does not provide something obvious, but the default should be to use Carbon classes, tokens, and grid
- Do NOT introduce any other UI libraries such as Material UI, Chakra, Bootstrap, or DaisyUI

  https://github.com/carbon-design-system/carbon
- Official React tutorial app (Next.js 13 + Carbon)

- Carbon Web Components library (for understanding how they structure layout, grid, and tokens, even though we are using React here)  
  https://github.com/carbon-design-system/carbon-web-components
- Vite based Web Components tutorial repo

High level migration plan
1. Scan the current React + Vite + Tailwind + shadcn frontend to understand:
   - Page structure and routes
   - Existing layout patterns
   - Components that come from shadcn/ui
   - Areas that are heavily styled with Tailwind

2. Set up Carbon in this Vite project:
   - Add the necessary packages:
     - `@carbon/react`
     - `@carbon/styles`
     - Any additional Carbon packages you deem useful, for example `@carbon/themes` and `@carbon/icons`
   - Configure Sass support in Vite if it is not already set up, since Carbon styles are authored in Sass
   - Create a main Sass entry file (for example `src/styles/carbon.scss`) that:
     - Imports Carbon reset styles
     - Imports the selected Carbon theme (for example Gray 10 or Gray 90)
     - Imports any component styles as recommended by Carbon
   - Ensure the Sass entry is imported at the root of the React app (for example in `main.tsx` or `App.tsx`)

3. Carbon theme and tokens
   - Choose a Carbon theme that fits a modern product website (for example Gray 10 for a light theme or Gray 90 for a dark theme) and apply it consistently
   - Use Carbon design tokens for:
     - Color (interactive, background, text, borders)
     - Typography (IBM Plex families, sizes, weights)
     - Spacing and layout scale
   - Replace hard coded colors, fonts, and spacing in Tailwind config and custom CSS with Carbon tokens wherever possible
   - Keep only minimal Tailwind tokens where absolutely necessary, but prefer Carbon tokens from `@carbon/themes`, `@carbon/layout`, and `@carbon/type`

4. Layout and structure
   - Implement a global layout using Carbon components and grid:
     - Use the Carbon UI Shell if appropriate:
       - `Header`, `HeaderName`, `HeaderNav`, `HeaderMenu`, `SideNav`, `SideNavItems`
     - Use the Carbon grid system instead of custom Tailwind flex and grid layouts:
       - Replace large layout structures with `Grid` and `Column` from Carbon
       - Respect Carbon responsive breakpoints
   - Create dedicated layout components, for example:
     - `AppShell` or `MainLayout` using UI Shell and Carbon grid for main content
   - Ensure the layout is fully responsive across mobile, tablet, and desktop using Carbon breakpoints

5. Replace shadcn and Tailwind based UI with Carbon components
   For every visible piece of UI, replace the existing implementation with the closest Carbon component while keeping behavior:

   - Buttons
     - Replace all button implementations (including shadcn `Button`) with Carbon `Button`
     - Use primary, secondary, tertiary, ghost, and danger variants according to Carbon usage guidelines
   - Inputs and forms
     - Replace all form fields with Carbon equivalents:
       - `TextInput`, `TextArea`, `NumberInput`, `Select`, `Dropdown`, `ComboBox`, `Checkbox`, `RadioButtonGroup`, `Toggle`
     - Use proper labels, helper text, and validation text patterns from Carbon
   - Navigation
     - Use UI Shell navigation components instead of custom navbars or shadcn navigation
     - Use `Breadcrumb` for hierarchical pages
   - Lists and data display
     - Replace custom tables and shadcn tables with Carbon `DataTable` or `DataTableSkeleton` where needed
     - Use `StructuredList`, `Accordion`, `Tabs`, `Tile`, `Tag`, `Card` like patterns for content groupings
   - Status and feedback
     - Replace any custom alerts, toasts, or banners with Carbon `InlineNotification` and `ToastNotification`
     - Use `Loading` and `Skeleton` components for loading states
   - Overlay and dialogs
     - Replace modal dialogs with Carbon `Modal` and confirm dialogs with `ComposedModal` patterns if appropriate
   - Icons
     - Remove third party icons and shadcn icons
     - Use Carbon icons imported from `@carbon/icons` or from the icon exports in `@carbon/react`
     - Choose icons that align with Carbon icon guidelines

6. Tailwind and shadcn removal or minimization
   - Gradually remove shadcn/ui:
     - Replace all shadcn components (buttons, dialogs, inputs, popovers, dropdowns, etc.) with Carbon components
     - Remove the shadcn provider and config once no shadcn components remain
   - Tailwind:
     - Remove Tailwind classes where Carbon already provides the layout, spacing, and typography
     - It is ok to keep a small amount of Tailwind utilities for very minor layout tweaks or spacing that is hard to express otherwise, but the main visual language and spacing should come from Carbon tokens and grid
     - If advisable, trim the Tailwind config to a minimal setup or remove it entirely once Carbon fully covers the UI

7. Code organization
   - Use a clean folder structure, for example:
     - `src/components/carbon` for shared Carbon based components or wrappers
     - `src/layouts` for shell and layout components
     - `src/pages` or `src/routes` for route level components if the app uses a router
   - Create small wrapper components for repeated Carbon patterns, for example:
     - `PrimaryButton`, `PageHeader`, `FormSection`, `CardWithTitle`
   - Document any non obvious usage of Carbon components with short comments

8. Accessibility
   - Ensure all interactive components preserve or improve accessibility:
     - Keyboard navigation
     - Focus states
     - aria labels and roles where required
   - Rely on Carbon defaults for accessible labels where possible and only add custom attributes when required

9. Express backend
   - Do not modify or remove the Express backend API
   - Keep all existing API calls and data fetching logic
   - If refactoring the frontend data layer, keep the same endpoints and request payloads or update them in a central place with clear comments

10. Final deliverables
   - A running React + Vite app where:
     - Carbon components are used consistently across the UI
     - shadcn/ui is no longer used
     - Tailwind is either minimal or removed, with styling handled primarily by Carbon
   - Clean and well structured TypeScript code
   - A brief summary comment in the README or in a top level file that explains:
     - Which Carbon theme is used
     - Where the main layout and Carbon configuration live
     - How to add new pages or components while staying aligned with Carbon
